§ 02 · Bug archaeology

Seven bugs that left traces

Three of them required GDB or a second-pass code review to find. The bugs are ordered by how long they took to locate, not by phase. Bug 02 produced the largest structural payoff: eliminating raw addresses removed twenty-six MISRA Rule 11.4 findings and an entire class of future typos.

01
IWDG write order
SymptomChip resets before main() reaches its first UART byte. Silent. No fault handler fires.
Root causeIWDG_KR requires 0xCCCC first, then 0x5555. Reversed, the LSI clock starts after the unlock window closes and the PVU/RVU sync flags never clear. The spin-loop while(IWDG->SR != 0) hangs forever. RM0351 §34.3.
Found withGDB single-step at power-on. No other symptom was visible.
InsightST's reference documents the sequence but not the failure mode. The LSI enable (RCC_CSR_LSION) and LSIRDY wait must also precede all IWDG config.
02
USART2_BRR offset typo
SymptomNo UART output. BRR reads back zero every time it is written. Four hours.
Root causeTyped 0x4000440C (CR3) instead of 0x40004408 (BRR). One hex digit difference, invisible at a glance.
Found withRegister dump in GDB confirming that CR3 was changing, not BRR.
InsightThe real fix was not the one-character correction. Every peripheral migrated to CMSIS struct access (USART2->BRR), after which twenty-six Rule 11.4 MISRA findings became obsolete and no raw address remained in the codebase.
03
Timeout post-decrement underflow
SymptomI2C error-recovery paths were dead code. A stuck bus never returned an error to the caller.
Root causewhile (!timeout) { ...; timeout--; } with an unsigned counter: the first iteration sets timeout to UINT_MAX on decrement because the condition !timeout is false at any nonzero value. The guard if (!timeout) below it was always false.
Found withMulti-pass code review. The happy path masked it entirely.
Fixwhile (timeout > 0U) { ...; timeout--; } with explicit if (timeout == 0U) guards. All error paths now reachable.
04
busy_delay_ms blocking the scheduler
SymptomUART output jitter, 150 ms stall visible at task startup. Sensor not blocking; scheduler itself was blocked.
Root causemax30102.c called busy_delay_ms(150) inside task_sensor for the MAX30102 reset. A CPU spin inside a task holds the scheduler for the full duration.
FixvTaskDelay(pdMS_TO_TICKS(150)). The init moved into task_sensor so it runs after vTaskStartScheduler(), where vTaskDelay is safe.
InsightAny code that runs inside a task after the scheduler starts cannot use busy-wait timing. The scheduler owns the CPU between yields.
05
TMP117 conversion formula
SymptomTemperature readings 0.05 °C low across the ambient range. Systematic, not noise.
Root causeraw × 78 / 1000 approximates 0.078, not the true 0.078125 °C/LSB.
Fix(raw × 625) / 8000: integer math, no float, exact conversion. Verified 25.3 °C ambient on hardware.
InsightThe TMP117 datasheet states 7.8125 m°C/LSB. The numerator 625 and denominator 8000 are the smallest integers that represent that ratio exactly.
06
Negative temperature sign placement
SymptomTemperatures below zero printed as 25.-3°C rather than -25.3°C.
Root causeThe sign character was emitted after the integer part, before the decimal digit, in the integer formatter.
FixTwo-line correction in uart.c: check sign before printing the whole-number part.
07
BPM refractory period correct by coincidence
SymptomNone at configTICK_RATE_HZ=1000. Would produce wrong peak rejection at any other tick rate.
Root causerefractory_ticks = 333 happened to equal 333 ms only because the tick rate was 1000 Hz. The constant was treated as milliseconds without conversion.
FixpdMS_TO_TICKS(333) corrected the unit conversion. BPM_REFRACTORY_MS was subsequently increased to 600 ms; current firmware uses pdMS_TO_TICKS(600U).
InsightCorrect-by-coincidence is a latent bug. The code looked right, passed all tests, and would have failed silently on a different board configuration.
Bug 02 produced the largest payoff. Eliminating raw volatile pointer casts removed twenty-six Rule 11.4 findings from the MISRA analysis and an entire class of future offset typos. Standard Cppcheck now returns zero findings on all nine project source files.
§ 03 · Architecture

Three layers, two tasks

Two tasks, one queue, one watchdog. task_sensor reads TMP117 and MAX30102, runs the 4-sample moving average and dynamic-threshold BPM detector, writes one sample to a depth-1 queue with xQueueOverwrite, kicks the IWDG, and sleeps 500 ms. task_uart blocks on xQueueReceive, formats, and transmits. The queue depth is one on purpose: UART is slower than the sensor, and stale readings are meant to be dropped.

Application
task_sensor (priority 2): sensors → filter → BPM → xQueueOverwrite → IWDG kick → vTaskDelay(500)
task_uart (priority 1): xQueueReceive blocks → format → UART TX

Processing
Moving average: 4-sample circular buffer, O(1) update, O(1) query, partial-window division until buffer fills.
BPM: dynamic threshold (midpoint of rolling min/max, 16-sample history), rising-edge, 600 ms refractory. Unsigned subtraction handles tick counter wraparound by design.
filter.c · bpm.c
↓ calls only the layer below
Driver
CMSIS struct access only. No raw hex addresses. No calls upward.
USART2 · I2C1 (100 kHz, PB8/PB9 open-drain) · TMP117 · MAX30102 · IWDG (LSI oscillator + write-order sequence)
uart · i2c · tmp117 · max30102 · iwdg

Authored: uart.c · i2c.c · tmp117.c · max30102.c · iwdg.c · filter.c · bpm.c · main.c · tasks_vitals.c (nine files, 692 lines total). syscalls.c / sysmem.c are IDE-generated newlib stubs for arm-none-eabi, retained unchanged. Application code allocates zero bytes at runtime. FreeRTOS heap_4 is the only allocator, used by the kernel for task stacks and the queue at startup. After vTaskStartScheduler(), the heap is frozen.

Two things not in any tutorial
1 CPACR must enable CP10 and CP11 before vTaskStartScheduler(). Set it after, and the first floating-point operation inside a task corrupts the saved context. The crash is delayed and looks like data corruption.
2 FreeRTOS owns SysTick_Handler. After the scheduler starts, application code reading SysTick->VAL for timing is reading a register the kernel is actively writing. bpm.c uses xTaskGetTickCount(). Unsigned subtraction handles the 32-bit tick counter wraparound without a branch.
§ 04 · Register map

CMSIS access

All access through stm32l476xx.h (STMicroelectronics/cmsis-device-l4). The compiler resolves struct offsets from RM0351. grep -r "0x4[0-9A-F]{7}" firmware/Core/Src/ returns zero matches.

Register CMSIS access Bits / value Note
RCCAHB2ENR RCC->AHB2ENR GPIOAEN · GPIOBEN Clock enable before any GPIO access
RCCAPB1ENR1 RCC->APB1ENR1 bit 17 = USART2 · bit 21 = I2C1
RCCCSR RCC->CSR LSION · LSIRDY LSI must be ready before IWDG config (see bug 01)
GPIOA MODER GPIOA->MODER bits 5:4 = PA2 AF mode 10 = AF, AF7 = USART2_TX via AFR[0]
GPIOB OTYPER GPIOB->OTYPER bits 8,9 = PB8, PB9 open-drain I2C requires open-drain on both SDA and SCL
GPIOB AFR[1] GPIOB->AFR[1] PB8, PB9 → AF4 = I2C1 AFRH register
USRT2 BRR USART2->BRR 417 = 9600 baud at 4 MHz MSI Bug 02: was 0x4000440C (CR3). CMSIS eliminates the risk.
USRT2 ISR USART2->ISR USART_ISR_TXE = bit 7 Poll before each TDR write
I2C1TIMINGR I2C1->TIMINGR 0x00100D14 100 kHz standard mode at 4 MHz MSI
IWDGKR IWDG->KR 0xCCCC start · 0x5555 unlock · 0xAAAA kick 0xCCCC before 0x5555. Order is not optional (bug 01).
